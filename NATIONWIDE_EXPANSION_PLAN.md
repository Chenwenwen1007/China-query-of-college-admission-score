# 全国高考录取数据库扩容计划

## 目标

把当前仅覆盖：

- 湖南
- 2023-2025
- 物化生
- 300-750 分（当前配置范围）

的专项数据库，逐步扩容为：

- 全国多省
- 可配置年份
- 可配置分数范围（先扩到 300-750）
- 可配置科类 / 选科组合
- 支持学校、专业、专业组、趋势、导出和复核

## 先决判断

这个扩容不能“一次性全国开抓”，原因有三个：

1. 各省公开文件格式差异极大。
2. 新高考省份和传统文理分科省份字段体系不同。
3. 当前库里“专业最低录取分”还没有独立真实来源，只有专业组投档线。

所以必须分阶段推进。

## 总体策略

先把当前湖南方案做成“省份可配置的 ETL 框架”，再按省份批量接入。

## 第一阶段当前进度

已完成：

1. 新增 `configs/provinces/hunan.json`，把省份代码、分数范围、官方源、数据库文件名抽成配置。
2. 新增 `province_config.py`，让 ETL 脚本和前端服务共享同一套省份配置加载逻辑。
3. `build_hunan_admissions_db.py` 已切到配置驱动的省份入口，不再把湖南配置散落写死在脚本顶部。
4. `frontend_server.py` 已支持 `--province hunan`，为后续切换到更多省份预留运行入口。
5. 湖南库已完成 `300-750` 配置分段重建，当前重建结果为 `10735` 个投档组、`47834` 条可报专业。

第一阶段未完成部分：

1. 把更多湖南专属常量继续抽离，减少脚本内部“湖南字段名/湖南批次语义”的隐式耦合。
2. 为后续更多省份接入补充断点续跑、分批构建和质量校验。
3. 设计跨省统一的字段标准，尤其是批次名、专业组标识、选科表达式和专业最低分真实性标记。

执行顺序建议：

1. 先把湖南 `300-750` 扩容后的 ETL 和校验流程稳定下来
2. 再做 `2-3` 个试点省份
3. 试点稳定后再扩到更多省份
4. 最后再做全国统一前端体验

## 阶段 0：数据定义与真实性边界

### 要先明确的四类数据

1. 学校 / 专业组投档线
2. 学校招生计划
3. 逐专业最低录取分
4. 选科 / 科类规则

### 当前真实情况

- 当前库已经稳定拥有：
  - 学校 / 专业组投档线
  - 招生计划
  - 选科规则
- 当前库还没有稳定拥有：
  - 全国范围的真实逐专业最低录取分

### 本阶段输出

- 一份字段定义文档
- 一份“哪些字段是官方真实字段、哪些字段是推导字段”的说明
- 一份“逐专业最低分”数据源候选清单

## 阶段 1：数据库与脚本泛化

### 目标

把当前最初写死“湖南 + 43.json + 物化生 + 435-490”的脚本改造成可配置框架。

### 必改项

- `sources` 表增加：
  - `province_code`
  - `province_name`
  - `subject_scheme`
  - `data_scope`
- `admission_groups`、`eligible_majors`、`plan_major_details` 增加省份维度
- 脚本拆分为：
  - 下载层
  - 解析层
  - 标准化层
  - 匹配层
  - 校验层

### 输出

- `configs/provinces/*.json`
- 通用构建脚本
- 通用表结构升级脚本

## 阶段 2：湖南先扩到 300-750 分

### 目标

在不增加省份复杂度的前提下，先验证全分段是否稳定。

### 风险点

- 记录数会明显增长
- 前端默认查询结果会变多
- 导出体积会增大
- 索引和接口性能需要重新评估

### 必做事项

- 取消 `435-490` 限制，改成参数化分数范围
- 默认前端查询仍保留分页 / 限量
- 给 SQLite 增加更细的组合索引
- 验证全量重建时间

### 输出

- 湖南 `300-750` 全分段数据库
- 性能报告
- 未匹配原因新统计

## 阶段 3：试点省份接入

### 推荐先做的省份类型

1. 新高考且公开文件较规范的省份
2. 与湖南类似的“专业组”结构省份
3. 再补传统文理分科省份

### 原因

这样能先复用现有专业组匹配逻辑，再逐步兼容文理分科差异。

### 每个新省份要完成的内容

1. 找到官方投档线源
2. 找到可稳定下载的原始文件
3. 建立字段映射
4. 确认科类/选科解释规则
5. 验证 `gaokao.cn` 对应省份计划是否稳定
6. 输出未匹配清单

## 阶段 4：逐专业最低录取分数据专项

### 这是全国化前最关键的一道门槛

因为你已经明确指出：

- 现在页面里展示的是学校专业组线
- 不是逐专业最低录取分

所以这部分必须单独立项。

### 需要做的事

1. 找到能稳定提供“学校-专业-年份-最低分”的真实来源
2. 核实它是否覆盖多省、多年
3. 确认它与当前学校、专业、专业组能否准确关联
4. 增加单独的数据表，例如：
   - `major_score_lines`
5. 做来源交叉校验

### 通过标准

- 至少在试点省份中，逐专业最低分字段可批量验证
- 不满足验证条件时，不对外宣称“真实专业最低分”

## 阶段 5：全国批量接入

### 推进方式

按省份批次推进，而不是一省一省完全手工散做。

### 推荐批次

1. 湖南模板完成
2. 2-3 个新高考试点省
3. 新高考省批量
4. 传统文理分科省批量
5. 复杂专项/特殊批次清理

## 阶段 6：校验与审计体系

### 必须有的校验能力

- 每条来源文件可追溯
- 每次构建有运行日志
- 每个省有未匹配明细
- 每个省有抽样校验 SQL
- 前端字段清楚区分：
  - 官方字段
  - 推导字段
  - 当前仅可见字段

### 建议新增

- `audit_reports` 表
- 自动生成每省校验报告
- 高风险映射人工确认清单

## 阶段 7：前端全国化改造

### 前端需要新增

- 省份切换
- 科类 / 选科切换
- 分数范围切换
- 批次切换
- 真实专业最低分与专业组投档线分开展示
- 来源标签
- 校验状态提示

## 阶段 8：资源与时间预估

### 工作量判断

这是一个“数据工程 + 抓取适配 + 校验工程 + 产品工程”混合项目，不是简单脚本扩几行。

### 主要成本

- 各省官方源摸排
- 文件格式适配
- 学校/专业映射维护
- 校验和复核
- 真实专业最低分专项

## 建议的立即下一步

最合理的顺序是：

1. 先修正当前前端字段表达，明确区分专业组线与专业线
2. 先把湖南扩到 `300-750`
3. 同时启动“真实专业最低分数据源”专项调研
4. 湖南稳定后，再做全国试点省方案

## 当前结论

全国扩容可以做，但必须遵守两个原则：

1. 先框架化，再扩省份
2. 先搞清楚真实专业最低分数据源，再对外展示“专业录取分”
